# S-003: Anonymous Response Submission with Real-Time Wall

## User Story

As a **team member**, I want to **type an anonymous response to the icebreaker question and see everyone's responses appear on a shared wall in real-time** so that **I can participate without speaking up and feel the team's energy building**.

## Context

This is the core "shy people" feature. Instead of discussing the icebreaker question verbally, everyone types responses anonymously. Responses appear on a shared wall — like sticky notes on a whiteboard, but digital and real-time. You see "someone wrote..." appear one by one. The wall creates a shared experience without requiring anyone to speak.

## Acceptance Criteria

- [ ] During icebreaker phase, a text input appears below the question for all participants
- [ ] Participants can submit short text responses (max 280 characters — tweet-length, not essays)
- [ ] Responses are **always anonymous** — no author attribution shown (author stored in DB for moderation)
- [ ] Submitted responses appear on the shared wall for ALL participants in real-time via WebSocket
- [ ] New responses animate onto the wall (entrance animation — see S-004 for specifics)
- [ ] Wall shows all responses from the current icebreaker session (scoped to this board + icebreaker_id)
- [ ] Participants can submit multiple responses (no limit, but rate-limited to 1 per 2 seconds)
- [ ] Empty or whitespace-only responses are rejected
- [ ] Input clears after successful submission
- [ ] Response count is visible (e.g., "12 responses" — gives a sense of participation)
- [ ] Wall is scrollable if responses overflow the viewport
- [ ] Facilitator can delete inappropriate responses (soft delete, disappears for everyone)

## Implementation Notes

### Database

New migration — `icebreaker_responses` table:

```sql
CREATE TABLE icebreaker_responses (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  board_id    UUID NOT NULL REFERENCES boards(id) ON DELETE CASCADE,
  icebreaker_id UUID NOT NULL REFERENCES icebreakers(id) ON DELETE CASCADE,
  author_id   UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  content     TEXT NOT NULL,
  deleted_at  TIMESTAMPTZ,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_icebreaker_responses_board ON icebreaker_responses(board_id, icebreaker_id);
```

### Backend

- `POST /api/v1/boards/:boardId/icebreaker/responses` — submit a response
  - Validates: board is in icebreaker phase, user is team member, content is 1-280 chars
  - Stores response with `author_id` (for moderation) but never exposes author to clients
  - Broadcasts `icebreaker_response_added` via WebSocket: `{ id, content, created_at }` (no author)
  - Rate limit: 1 response per 2 seconds per user

- `GET /api/v1/boards/:boardId/icebreaker/responses` — get all responses for current icebreaker
  - Returns responses where `deleted_at IS NULL`
  - Ordered by `created_at ASC`
  - Never includes `author_id` in response

- `DELETE /api/v1/boards/:boardId/icebreaker/responses/:responseId` — facilitator-only soft delete
  - Sets `deleted_at = NOW()`
  - Broadcasts `icebreaker_response_removed` via WebSocket: `{ id }`

### Frontend

- New Zustand slice or store section for icebreaker responses
- Text input: single line, placeholder "Type your answer...", submit on Enter or button click
- Wall layout: CSS grid or flexbox wrapping cards, responsive
- Each response card: white card with content text, subtle shadow, rounded corners
- Response cards don't show who wrote them — ever
- WebSocket handlers: `icebreaker_response_added` → add to store, `icebreaker_response_removed` → remove from store
- Facilitator sees a small "X" on each response card for deletion
- Participation counter: "12 responses" shown near the top

### Design

The wall is the main area. Layout from top to bottom:
1. Question (from S-002) — large, centered
2. Response wall — takes up most of the screen
3. Input bar — fixed at bottom (like a chat input)
