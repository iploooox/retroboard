# S-005: Emoji Reactions on Wall Responses

## User Story

As a **team member**, I want to **react to responses on the wall with emojis** so that **I can engage with my teammates' answers without typing — just one tap to say "that's hilarious" or "same here"**.

## Context

Reactions are the lowest-friction form of participation. Even the shyest person will tap a 🔥 on a funny response. This story adds per-response emoji reactions — tap an emoji under a response card, and the reaction count appears for everyone. This is the "like button" layer. The full reaction rain animation is S-006.

## Acceptance Criteria

- [ ] Each response card on the wall has a reaction bar below it
- [ ] Reaction bar shows 6 emoji options: 😂 🔥 ❤️ 🎯 👏 💀
- [ ] Tapping an emoji adds your reaction to that response (one reaction per emoji per user per response)
- [ ] Tapping the same emoji again removes your reaction (toggle behavior)
- [ ] Reaction counts are visible on the card (e.g., "🔥 3 😂 5")
- [ ] Only emojis with count > 0 show counts (don't show "🔥 0")
- [ ] Reactions are synced in real-time via WebSocket — when someone reacts, all participants see the count update
- [ ] Reactions are **anonymous** — you can see counts but not who reacted
- [ ] Reaction bar is compact — doesn't dominate the card, appears on hover/tap (mobile: always visible)
- [ ] Optimistic UI: count updates instantly on tap, rolls back on error
- [ ] When a response card is deleted, its reactions are cascade-deleted

## Implementation Notes

### Database

New migration — `icebreaker_response_reactions` table:

```sql
CREATE TABLE icebreaker_response_reactions (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  response_id   UUID NOT NULL REFERENCES icebreaker_responses(id) ON DELETE CASCADE,
  user_id       UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  emoji         VARCHAR(10) NOT NULL,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),

  CONSTRAINT uq_response_reaction UNIQUE (response_id, user_id, emoji),
  CONSTRAINT chk_valid_emoji CHECK (emoji IN ('laugh', 'fire', 'heart', 'bullseye', 'clap', 'skull'))
);

CREATE INDEX idx_response_reactions_response ON icebreaker_response_reactions(response_id);
```

Store emoji as short strings (`'fire'`, `'laugh'`) not actual unicode — cleaner for DB constraints and API.

### Backend

- `POST /api/v1/boards/:boardId/icebreaker/responses/:responseId/reactions` — add reaction
  - Body: `{ emoji: "fire" }`
  - Toggle: if reaction exists, delete it; if not, insert it
  - Returns: `{ action: "added" | "removed", emoji, count }`
  - Broadcasts `icebreaker_reaction_updated` via WebSocket: `{ responseId, emoji, count }`

- Response GET endpoint (S-003) includes reaction counts per response:
  ```json
  {
    "id": "...",
    "content": "Tokyo ramen",
    "reactions": { "fire": 3, "laugh": 5 },
    "myReactions": ["fire"]
  }
  ```
  - `myReactions` tells the client which emojis the current user has already reacted with (for toggle state)
  - Reaction counts use a single aggregate query, not N+1

### Frontend

- Reaction bar component: row of 6 emoji buttons below each response card
- Desktop: reaction bar appears on card hover (opacity transition)
- Mobile: reaction bar is always visible (compact, smaller emoji)
- Active state: emoji button is highlighted if user has reacted (e.g., subtle background color)
- Count badge: small number next to emoji, only shown if count > 0
- Optimistic update: increment count immediately, toggle highlight, revert on API error
- WebSocket handler: `icebreaker_reaction_updated` → update reaction count in store for specific response
