# S-015: Facilitation Phases (Write, Group, Vote, Discuss, Action)

## User Story

As a **facilitator**, I want to **guide the team through structured retro phases** so that **the retrospective follows a productive flow and each activity happens at the right time**.

## Acceptance Criteria

- [ ] Board has five facilitation phases: write, group, vote, discuss, action
- [ ] Facilitator can advance to the next phase or go back to a previous phase
- [ ] Current phase is visible to all participants in real-time
- [ ] During "write" phase: card creation is enabled, cards may be hidden from others (reveal later)
- [ ] During "group" phase: facilitator can group cards, card creation is disabled
- [ ] During "vote" phase: voting is enabled, card creation and grouping are disabled
- [ ] During "discuss" phase: all cards sorted by votes, discussion mode enabled, no new cards/votes
- [ ] During "action" phase: action item creation is enabled, board is in wrap-up mode
- [ ] Phase transitions are broadcast to all participants via WebSocket
- [ ] Phase restrictions are enforced server-side (API rejects disallowed operations for current phase)
- [ ] Board without facilitation phases enabled allows all operations freely (unstructured mode)

## Implementation Notes

- Add `current_phase` column to `boards` table (nullable, null = unstructured mode)
- Phase enum: 'write', 'group', 'vote', 'discuss', 'action'
- Phase restrictions matrix maps each phase to allowed/disallowed operations
- Server middleware checks current phase before allowing card/vote/group operations
- Phase transition event: `board:phase_changed`
