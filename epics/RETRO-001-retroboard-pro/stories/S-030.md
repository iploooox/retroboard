# S-030: Keyboard Shortcuts and Accessibility

## User Story

As a **user**, I want to **navigate and interact with the retro board using keyboard shortcuts** so that **I can work efficiently and the application is accessible to users who rely on keyboard navigation**.

## Acceptance Criteria

- [ ] Keyboard shortcuts are available for common board actions
- [ ] Shortcut reference: N (new card), E (edit selected card), Delete (delete selected card), V (vote on selected card)
- [ ] Navigation: Tab/Shift+Tab to navigate between cards, Arrow keys to move between columns
- [ ] Facilitator shortcuts: L (lock/unlock), R (reveal), Space (next phase), T (start/stop timer)
- [ ] Shortcuts panel: ? key opens a shortcuts help overlay listing all available shortcuts
- [ ] All interactive elements are focusable and operable via keyboard
- [ ] Focus indicators are clearly visible (outline/highlight on focused elements)
- [ ] Screen reader announcements for dynamic content (card added, vote count changed, phase changed)
- [ ] ARIA labels and roles are applied to all custom components
- [ ] Color is not the only means of conveying information (patterns, icons, text labels used alongside)
- [ ] Board meets WCAG 2.1 AA compliance
- [ ] Keyboard shortcuts do not conflict with browser or screen reader defaults
- [ ] Shortcuts are disabled when user is typing in a text input

## Implementation Notes

- Use a keyboard shortcut library (e.g., hotkeys-js, react-hotkeys-hook)
- Implement a shortcut manager that registers/unregisters based on context (board, dialog, input)
- ARIA live regions for real-time updates (card added, votes changed)
- Focus management: trap focus in modals, return focus after modal close
- Test with screen readers (NVDA, VoiceOver) and keyboard-only navigation
- Consider reduced motion preference for animations
